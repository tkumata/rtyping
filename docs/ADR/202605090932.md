# ADR: GitHub Release ノート生成と Release 作成 job の分離

## 状況

`version-check.yml` は `Cargo.toml` の version 変更を検出し、Linux と macOS の matrix build 内で `softprops/action-gh-release` を実行していた。

この構成では OS 別 job が同じ tag / Release を並列に更新し、リリース本文は GitHub の `generate_release_notes` に依存するため、利用者向けに変更点と成果物を説明する情報が薄くなる。

## 決定

- `Cargo.toml` の package version を Release の source of truth として維持する。
- Release 前に `make check` と `make build` を実行する。
- OS 別 build job は成果物作成と artifact upload のみに限定する。
- 単一の release job で artifact を収集し、`RELEASE_NOTES.md` を生成する。
- `softprops/action-gh-release` は `body_path` と収集済み成果物を使い、1回だけ実行する。

## 結果

- Release 作成の重複実行を避けられる。
- リリースノートに version、変更履歴、成果物一覧を安定して含められる。
- 将来 `git-cliff` や `release-plz` を導入する場合も、release job の note generation 部分だけを差し替えられる。

## 検証

- `make check`
- `make build`
