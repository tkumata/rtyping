# 要件定義 (REQUIREMENTS)

## 概要

`rtyping` は Rust 製の TUI タイピングアプリであり、ローカル文生成と外部 API 文生成、設定保存、結果表示を一連の操作として提供する。

## 機能要件

1. 起動時にタイトルメニューを表示し、`Start Game`、`Practice Mode`、`Start Game via Google AI Studio`、`Start Game via Groq`、`Stats`、`Config` を選択できること。
2. 文字列生成元として `Local`、`Google AI Studio`、`Groq` をサポートすること。
3. タイトルメニューからゲーム開始した場合は、選択した項目に対応する生成元とモードで文字列生成を開始できること。
4. `Config` 画面では各プロバイダの `API URL`、`API key`、`Model` を編集できること。
5. `Config` 画面でゲーム設定（制限時間、テキスト量、音量周波数、サウンド有効化）を編集してファイル保存できること。
6. `Config` 画面では現在フォーカスしている入力欄にカーソルを表示し、左右キーで入力位置を移動できること。
7. 設定は起動時に読み込み、保存時は `API key` を平文で保存しないこと。
8. API 利用時に設定不足、通信失敗、復号失敗が発生してもアプリが異常終了せず、メッセージで状態を説明できること。
9. タイピング中は制限時間を計測し、完了またはタイムアウト時に結果画面へ遷移すること。
10. Config 画面でタイムアウトを `0` に設定した場合はタイムアップのない練習モードとして動作し、入力完了まで結果画面へ遷移しないこと。
11. タイトルメニューの `Practice Mode` 選択時はタイムアップのない練習モードとして開始できること。
12. タイピング中は strict 判定を行い、誤入力では現在位置を進めず、正しい文字のみ先へ進めること。
13. タイピング中に `Esc` を押した場合はメニュー画面へ戻れること。
14. 結果画面では総入力数、ミス数、正確率、経過時間、WPM を確認できること。
15. 正確率は `accuracy = (total_typed_count - incorrects) / total_typed_count * 100` で算出し、`total_typed_count = 0` の場合は `0.0%` として扱うこと。
16. Config 画面の `SoundEnabled` 設定に従い BGM と正答音を有効化すること。BGM はアプリ再起動後に反映され、キープレス音は設定変更後即時反映されること。
17. タイピング中は小さなグラフ領域に `ratatui` の線グラフ系ウィジェットを用いて WPM の推移をリアルタイム表示できること。
18. 結果画面では、タイピング中に記録した WPM 履歴の最終状態を固定グラフとして表示できること。
19. WPM グラフは各画面の主要テキストと重ならず、別領域として描画されること。
20. WPM 推移の記録はタイピング開始時に初期化し、ラウンド終了または中断時に次のラウンドへ持ち越さないこと。
21. Typing 画面では、端末カーソルを出題文字列上に表示せず、現在入力すべき文字を文字色と太字で示せること。
22. Typing 画面の現在入力すべき文字は `Yellow + Bold` で表示できること。
23. Typing 画面の未来の未入力文字は `Gray` で表示できること。
24. Typing 画面では、出題文字列が改行で折り返されても現在入力すべき文字の強調が実際の入力位置と一致すること。
25. Typing 画面の WPM グラフは、既定色を維持しつつ高い WPM の線分をオレンジで強調できること。
26. Typing 中に入力操作が発生していない区間は、最後の入力から 2 秒の猶予までは WPM を維持し、猶予経過後に 0 として表示すること。
27. Timed セッション完了時に WPM、正確率、ミス数、入力時間、生成元、モード、正解側ミス文字を履歴として保存できること。
28. Practice Mode の結果は成績履歴に保存しないこと。
29. 成績履歴は `~/.config/rtyping/history.json` に保存できること。
30. Result 画面では自己ベスト WPM、平均 WPM、平均正確率、直近10回の WPM 推移、よく間違える文字を確認できること。
31. Stats 画面では保存済み履歴の統計サマリを確認できること。
32. 履歴が存在しない場合でも Result / Stats 画面を表示できること。
33. 起動直後のタイトルメニューでは、現在選択中の項目を `▶︎` で示すこと。
34. Typing 画面の `Target Text` ブロックでは、出題本文の上に2行、出題本文の下に2行の空行を表示できること。
35. `Target Text` の出題本文が複数行に折り返されても、本文下の2行の空行を維持できること。
36. WPM Trend ブロックの枠線は、グラフ本体の線色とは独立して薄い黄色で表示できること。
37. `Config` 画面の文字入力は現在のカーソル位置へ挿入できること。
38. `Config` 画面の `Backspace` は現在のカーソル直前の文字を削除できること。
39. `Config` 画面の API key 欄は平文を表示せず、実文字数と同じ長さのマスクでカーソル位置を判別できること。
40. Google AI Studio と Groq による外部 API 生成では、リクエストごとに異なる variation seed を含むプロンプトを送信できること。
41. 外部 API 生成の最終的な出題文は、既存の `TextScale` から算出した目標文字数を超えないこと。
42. `Cargo.toml` の version 更新を起点に、GitHub Actions で自動リリースを実行できること。
43. 自動リリースでは OS 別ビルド成果物を単一の GitHub Release に添付できること。
44. 自動リリースでは GitHub 自動生成だけに依存せず、version、変更履歴、成果物一覧を含むリリースノートを作成できること。

## 非機能要件

1. 主要なユースケースと設定復元処理は `cargo test` で回帰確認できること。
2. UI 状態遷移とランタイム制御は、初見の保守担当者が数時間で追跡できる粒度に責務分割されていること。
3. `docs/` の記述は現行実装と矛盾しないこと。
4. 結果指標の追加時はゼロ除算や未入力終了時の表示崩れを発生させないこと。
5. 練習モードと strict 判定を追加する場合は、タイマー終了条件、入力進行条件、`Esc` の遷移先を同時に定義して挙動の曖昧さを残さないこと。
6. WPM グラフ追加時は画面が狭い場合でも各画面の主要情報の可読性を優先し、重なりを発生させないこと。
7. 結果画面へグラフを追加しても、既存の結果指標と終了導線が視認できる情報量に収めること。
8. `cargo clippy` の既知警告を残さず、少なくとも変更対象モジュールでは `items_after_test_module` のような構造警告を解消すること。
9. Typing 画面の現在位置表示を変更する場合は、入力中の可読性を優先し、出題文字列や WPM 表示と干渉しないこと。
10. Typing 画面では端末カーソルで細い記号を隠さず、現在入力すべき文字のスタイルで位置を示すこと。
11. Typing 画面の現在文字強調は、改行・折返し後の表示位置と一致すること。
12. エディタの TOML スキーマ検証で、Cargo が有効に扱う lint 設定を誤ってエラー表示しないこと。
13. 履歴ファイルが壊れていてもアプリを異常終了させず、警告を表示して空履歴として扱うこと。
14. タイトルメニューの選択マーカー変更は表示上の変更に限定し、メニュー選択順、入力操作、状態遷移へ影響させないこと。
15. `Target Text` ブロックの余白変更は表示レイアウトに限定し、入力判定、WPM グラフ、カーソル形状、カーソル位置の意味を変更しないこと。
16. WPM Trend ブロックの枠線色変更は表示上の変更に限定し、WPM 履歴、グラフ線色、高値強調、入力判定、画面遷移へ影響させないこと。
17. Config 画面の入力カーソル変更は設定保存形式を変更せず、既存の URL、モデル、API key、ゲーム設定の永続化互換性を維持すること。
18. 外部 API 生成の固定化回避は API 側の出力トークン上限を追加せず、文字数制御は既存の正規化処理に委ねること。
19. 自動リリースは Release 作成を matrix job 内で重複実行せず、単一 job で実行すること。
20. 自動リリース前には `make check` と `make build` に成功していること。

## 保守上の前提

1. `src/main.rs` は起動と終了処理を中心に保ち、イベントループや入力処理は別モジュールへ分離する。
2. 設定保存の互換処理や API 生成の失敗系は、挙動をテストで固定する。
3. 外部仕様を変えない内部改善でも、読み手が誤解する古い説明は更新対象とする。
4. ミス修正回数と実ミス数の分離は将来拡張候補とし、今回の変更では既存の `incorrects` の意味を変えない。
5. タイピング中の追加メトリクスは、表示値だけでなくセッション開始・終了時の初期化条件も併せて管理する。
6. リアルタイム表示で収集した履歴を結果画面へ再利用する場合は、別の集計値を導入せず同一データの表示責務だけを増やす。
7. Rust モジュール内のテストは原則としてファイル末尾へ配置し、テストモジュール以後に通常項目を置かない。
8. Cargo manifest の lint 設定は Cargo の解釈を優先し、エディタ向けの補助設定で実装時の lint 強度を下げない。
9. 履歴集計は Result と Stats で同じロジックを使用し、画面ごとに計算定義を分岐させない。
10. Release の source of truth は `Cargo.toml` の package version とし、タグ名は `v<version>` 形式に揃える。
