# 要件定義 (REQUIREMENTS)

## 概要

`rtyping` は Rust 製の TUI タイピングアプリであり、ローカル文生成と外部 API 文生成、設定保存、結果表示を一連の操作として提供する。

## 機能要件

1. 起動時にタイトルメニューを表示し、`Start Game`、`Practice Mode`、`Start Game via Google AI Studio`、`Start Game via Groq`、`Config` を選択できること。
2. 文字列生成元として `Local`、`Google AI Studio`、`Groq` をサポートすること。
3. `--google` と `--groq` は排他的であり、未指定時は `Local` を使うこと。
4. タイトルメニューからゲーム開始した場合は、選択した項目に対応する生成元とモードで文字列生成を開始できること。
5. `Config` 画面では各プロバイダの `API URL`、`API key`、`Model` を編集できること。
6. 設定は起動時に読み込み、保存時は `API key` を平文で保存しないこと。
7. API 利用時に設定不足、通信失敗、復号失敗が発生してもアプリが異常終了せず、メッセージで状態を説明できること。
8. タイピング中は制限時間を計測し、完了またはタイムアウト時に結果画面へ遷移すること。
9. `-t 0` または `--timeout 0` 指定時はタイムアップのない練習モードとして動作し、入力完了まで結果画面へ遷移しないこと。
10. タイトルメニューの `Practice Mode` 選択時は、CLI の制限時間指定にかかわらずタイムアップのない練習モードとして開始できること。
11. タイピング中は strict 判定を行い、誤入力では現在位置を進めず、正しい文字のみ先へ進めること。
12. タイピング中に `Esc` を押した場合はメニュー画面へ戻れること。
13. 結果画面では総入力数、ミス数、正確率、経過時間、WPM を確認できること。
14. 正確率は `accuracy = (total_typed_count - incorrects) / total_typed_count * 100` で算出し、`total_typed_count = 0` の場合は `0.0%` として扱うこと。
15. `--sound` 指定時のみ BGM と正答音を有効化すること。
16. タイピング中は小さなグラフ領域に `ratatui` の `Sparkline` ウィジェットを用いて WPM の推移をリアルタイム表示できること。
17. WPM グラフは出題中の文字列と重ならず、別領域として描画されること。
18. WPM 推移の記録はタイピング開始時に初期化し、ラウンド終了または中断時に次のラウンドへ持ち越さないこと。

## 非機能要件

1. 主要なユースケースと設定復元処理は `cargo test` で回帰確認できること。
2. UI 状態遷移とランタイム制御は、初見の保守担当者が数時間で追跡できる粒度に責務分割されていること。
3. `docs/` の記述は現行実装と矛盾しないこと。
4. 結果指標の追加時はゼロ除算や未入力終了時の表示崩れを発生させないこと。
5. 練習モードと strict 判定を追加する場合は、タイマー終了条件、入力進行条件、`Esc` の遷移先を同時に定義して挙動の曖昧さを残さないこと。
6. WPM グラフ追加時は画面が狭い場合でも出題文字列の可読性を優先し、重なりを発生させないこと。

## 保守上の前提

1. `src/main.rs` は起動と終了処理を中心に保ち、イベントループや入力処理は別モジュールへ分離する。
2. 設定保存の互換処理や API 生成の失敗系は、挙動をテストで固定する。
3. 外部仕様を変えない内部改善でも、読み手が誤解する古い説明は更新対象とする。
4. ミス修正回数と実ミス数の分離は将来拡張候補とし、今回の変更では既存の `incorrects` の意味を変えない。
5. タイピング中の追加メトリクスは、表示値だけでなくセッション開始・終了時の初期化条件も併せて管理する。
