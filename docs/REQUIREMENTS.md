# 要件定義 (REQUIREMENTS)

## 概要

`rtyping` は Rust 製の TUI タイピングアプリであり、ローカル文生成と外部 API 文生成、設定保存、結果表示を一連の操作として提供する。

## 機能要件

1. 起動時にタイトルメニューを表示し、`Start Game`、`Practice Mode`、`Start Game via Google AI Studio`、`Start Game via Groq`、`Config` を選択できること。
2. 文字列生成元として `Local`、`Google AI Studio`、`Groq` をサポートすること。
3. タイトルメニューからゲーム開始した場合は、選択した項目に対応する生成元とモードで文字列生成を開始できること。
4. `Config` 画面では各プロバイダの `API URL`、`API key`、`Model` を編集できること。
5. `Config` 画面でゲーム設定（制限時間、テキスト量、音量周波数、サウンド有効化）を編集してファイル保存できること。
6. 設定は起動時に読み込み、保存時は `API key` を平文で保存しないこと。
7. API 利用時に設定不足、通信失敗、復号失敗が発生してもアプリが異常終了せず、メッセージで状態を説明できること。
8. タイピング中は制限時間を計測し、完了またはタイムアウト時に結果画面へ遷移すること。
9. Config 画面でタイムアウトを `0` に設定した場合はタイムアップのない練習モードとして動作し、入力完了まで結果画面へ遷移しないこと。
10. タイトルメニューの `Practice Mode` 選択時はタイムアップのない練習モードとして開始できること。
11. タイピング中は strict 判定を行い、誤入力では現在位置を進めず、正しい文字のみ先へ進めること。
12. タイピング中に `Esc` を押した場合はメニュー画面へ戻れること。
13. 結果画面では総入力数、ミス数、正確率、経過時間、WPM を確認できること。
14. 正確率は `accuracy = (total_typed_count - incorrects) / total_typed_count * 100` で算出し、`total_typed_count = 0` の場合は `0.0%` として扱うこと。
15. Config 画面の `SoundEnabled` 設定に従い BGM と正答音を有効化すること。BGM はアプリ再起動後に反映され、キープレス音は設定変更後即時反映されること。
16. タイピング中は小さなグラフ領域に `ratatui` の `Sparkline` ウィジェットを用いて WPM の推移をリアルタイム表示できること。
17. 結果画面では、タイピング中に記録した WPM 履歴の最終状態を固定グラフとして表示できること。
18. WPM グラフは各画面の主要テキストと重ならず、別領域として描画されること。
19. WPM 推移の記録はタイピング開始時に初期化し、ラウンド終了または中断時に次のラウンドへ持ち越さないこと。
20. Typing 画面では、現在入力位置のカーソルを四角ではなく縦棒として表示できること。
21. Typing 画面の縦棒カーソルは、現在入力位置の文字の左側ではなく右側に表示できること。
22. Typing 画面では、出題文字列が改行で折り返されても現在入力位置のカーソルが実際の表示位置に一致し、行をまたぐたびにずれないこと。
23. Typing 画面のカーソル位置計算は、出題文字列の表示折り返しと同じ条件で行うこと。

## 非機能要件

1. 主要なユースケースと設定復元処理は `cargo test` で回帰確認できること。
2. UI 状態遷移とランタイム制御は、初見の保守担当者が数時間で追跡できる粒度に責務分割されていること。
3. `docs/` の記述は現行実装と矛盾しないこと。
4. 結果指標の追加時はゼロ除算や未入力終了時の表示崩れを発生させないこと。
5. 練習モードと strict 判定を追加する場合は、タイマー終了条件、入力進行条件、`Esc` の遷移先を同時に定義して挙動の曖昧さを残さないこと。
6. WPM グラフ追加時は画面が狭い場合でも各画面の主要情報の可読性を優先し、重なりを発生させないこと。
7. 結果画面へグラフを追加しても、既存の結果指標と終了導線が視認できる情報量に収めること。
8. `cargo clippy` の既知警告を残さず、少なくとも変更対象モジュールでは `items_after_test_module` のような構造警告を解消すること。
9. Typing 画面のカーソル形状を変更する場合は、入力中の可読性を優先し、出題文字列や WPM 表示と干渉しないこと。
10. Typing 画面の縦棒カーソルは、現在入力位置の文字の右側に置き、文字列の読み順を妨げないこと。
11. Typing 画面のカーソル位置計算は、改行・折返し境界で誤差を蓄積させず、再描画後の表示位置と一致すること。

## 保守上の前提

1. `src/main.rs` は起動と終了処理を中心に保ち、イベントループや入力処理は別モジュールへ分離する。
2. 設定保存の互換処理や API 生成の失敗系は、挙動をテストで固定する。
3. 外部仕様を変えない内部改善でも、読み手が誤解する古い説明は更新対象とする。
4. ミス修正回数と実ミス数の分離は将来拡張候補とし、今回の変更では既存の `incorrects` の意味を変えない。
5. タイピング中の追加メトリクスは、表示値だけでなくセッション開始・終了時の初期化条件も併せて管理する。
6. リアルタイム表示で収集した履歴を結果画面へ再利用する場合は、別の集計値を導入せず同一データの表示責務だけを増やす。
7. Rust モジュール内のテストは原則としてファイル末尾へ配置し、テストモジュール以後に通常項目を置かない。
